HTML Accessibility

Introduction to Web Accessibility

Web accessibility means designing and developing websites that can be used by everyone, including people with disabilities. It's about creating an inclusive web that doesn't exclude anyone based on their abilities or the way they access the internet.

According to the World Health Organization, over 1 billion people worldwide live with some form of disability. This represents about 15% of the global population. By making your websites accessible, you're not only doing the right thing ethically, but you're also expanding your potential audience significantly.

Key Benefits of Accessible Websites

  • Inclusivity: Ensures all users can access and interact with your content
  • Improved User Experience: Accessibility improvements often benefit all users
  • SEO Advantages: Many accessibility practices improve search engine rankings
  • Legal Compliance: Meets requirements of laws like ADA, Section 508, and AODA
  • Broader Audience: Reaches the 15% of the population with disabilities
  • Better Code Quality: Accessible code is often cleaner and more maintainable
  • Positive Brand Image: Demonstrates social responsibility and inclusivity

Understanding Different Types of Disabilities

To create truly accessible websites, it's important to understand the different types of disabilities and how they affect web usage:

Disability Type Web Usage Challenges Assistive Technologies Key Accessibility Considerations
Visual
(Blindness, low vision, color blindness)		 Cannot see images, videos, or text; difficulty with color-dependent information Screen readers, screen magnifiers, braille displays, high-contrast modes Alt text for images, keyboard navigation, proper heading structure, sufficient color contrast
Auditory
(Deafness, hard of hearing)		 Cannot hear audio content or alerts Captions, transcripts, visual alternatives for audio cues Captions for videos, transcripts for audio, visual indicators for sounds
Motor
(Limited dexterity, paralysis, tremors)		 Difficulty using mouse, clicking small targets, or performing precise movements Adaptive keyboards, mouth sticks, eye tracking, voice recognition Keyboard accessibility, large click targets, reduced motion requirements
Cognitive
(Learning disabilities, ADHD, autism, memory issues)		 Difficulty understanding complex content, remembering steps, or focusing Text-to-speech, reading tools, simplified interfaces Clear language, consistent navigation, minimal distractions, sufficient time to complete tasks
Seizure Disorders
(Photosensitive epilepsy)		 Risk of seizures from flashing content or certain visual patterns Content filters, reduced motion settings Avoiding flashing content, providing warnings, respecting reduced motion preferences

Web Content Accessibility Guidelines (WCAG)

The Web Content Accessibility Guidelines (WCAG) are the internationally recognized standards for web accessibility. They are organized around four main principles, often referred to as POUR:

  • Perceivable: Information and user interface components must be presentable to users in ways they can perceive
  • Operable: User interface components and navigation must be operable
  • Understandable: Information and the operation of the user interface must be understandable
  • Robust: Content must be robust enough to be interpreted reliably by a wide variety of user agents, including assistive technologies

WCAG has three levels of conformance:

  • Level A: The minimum level of accessibility; addresses the most basic accessibility features
  • Level AA: The level that most organizations aim for; addresses the most common barriers for disabled users
  • Level AAA: The highest level of accessibility; provides enhanced accessibility for specific user groups

In this lesson, we'll focus on HTML-specific techniques to meet WCAG requirements, primarily at the AA level, which is considered the standard for most websites.

Semantic HTML for Accessibility

Using semantic HTML is one of the most important aspects of creating accessible websites. Semantic HTML provides meaning to your content, which helps assistive technologies understand and navigate your page.

Document Structure

A well-structured HTML document provides a solid foundation for accessibility:

<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>Accessible Page Title</title>
    <meta name="description" content="Clear description of the page content">
</head>
<body>
    <header>
        <nav>
            <!-- Navigation content -->
        </nav>
    </header>
    
    <main>
        <article>
            <h1>Main Page Heading</h1>
            <section>
                <h2>Section Heading</h2>
                <p>Section content...</p>
            </section>
        </article>
        
        <aside>
            <h2>Related Information</h2>
            <p>Sidebar content...</p>
        </aside>
    </main>
    
    <footer>
        <p>Footer content...</p>
    </footer>
</body>
</html>

Key elements for accessible document structure:

  • <html lang="en">: Specifies the language of the document, helping screen readers pronounce content correctly
  • <title>: Provides a descriptive page title that appears in browser tabs and is announced by screen readers
  • <header>, <nav>, <main>, <article>, <section>, <aside>, <footer>: Semantic elements that define the purpose of different page regions

Heading Hierarchy

Proper heading structure is crucial for accessibility as it helps screen reader users navigate your content:

<!-- Good heading hierarchy -->
<h1>Main Page Title</h1>
<section>
    <h2>Section Title</h2>
    <p>Section content...</p>
    
    <h3>Subsection Title</h3>
    <p>Subsection content...</p>
    
    <h3>Another Subsection Title</h3>
    <p>More subsection content...</p>
</section>

<section>
    <h2>Another Section Title</h2>
    <p>More section content...</p>
</section>

<!-- Bad heading hierarchy (avoid this) -->
<h3>Page Title</h3> <!-- Should be h1 -->
<h1>Section Title</h1> <!-- Should be h2 -->
<h4>Subsection Title</h4> <!-- Should be h3 -->

Best practices for heading hierarchy:

  • Use only one <h1> per page, representing the main page title
  • Don't skip heading levels (e.g., don't go from <h2> to <h4>)
  • Use headings to create a logical outline of your content
  • Don't use headings just for styling; they should represent actual section titles

Text-Level Semantics

Use appropriate elements to mark up text content:

<!-- Emphasis and importance -->
<p>This is <em>emphasized</em> text.</p>
<p>This is <strong>important</strong> text.</p>

<!-- Abbreviations -->
<p>The <abbr title="World Health Organization">WHO</abbr> provides health guidelines.</p>

<!-- Time and dates -->
<p>The event starts on <time datetime="2025-05-15T19:00">May 15, 2025 at 7:00 PM</time>.</p>

<!-- Quotes -->
<blockquote>
    <p>The Web is designed to work for all people, whatever their hardware, software, language, location, or ability.</p>
    <cite>Tim Berners-Lee, W3C Director and inventor of the World Wide Web</cite>
</blockquote>

<!-- Code -->
<p>Use the <code>alt</code> attribute to provide text alternatives for images.</p>

<!-- Definition -->
<p><dfn>Web accessibility</dfn> means that websites, tools, and technologies are designed and developed so that people with disabilities can use them.</p>

Semantic HTML vs. Non-Semantic HTML

Compare these two approaches to creating a navigation menu:

Non-Semantic Approach:

<div class="navigation">
    <div class="nav-item"><a href="/">Home</a></div>
    <div class="nav-item"><a href="/about">About</a></div>
    <div class="nav-item"><a href="/contact">Contact</a></div>
</div>

Semantic Approach:

<nav aria-label="Main Navigation">
    <ul>
        <li><a href="/">Home</a></li>
        <li><a href="/about">About</a></li>
        <li><a href="/contact">Contact</a></li>
    </ul>
</nav>

The semantic approach:

  • Clearly identifies the purpose of the element (navigation)
  • Uses proper list markup for menu items
  • Includes an ARIA label for additional context
  • Is more accessible to screen reader users

Lists

Use appropriate list elements to structure related items:

<!-- Unordered list for items with no specific order -->
<ul>
    <li>Apples</li>
    <li>Oranges</li>
    <li>Bananas</li>
</ul>

<!-- Ordered list for sequential items -->
<ol>
    <li>Preheat the oven</li>
    <li>Mix the ingredients</li>
    <li>Bake for 20 minutes</li>
</ol>

<!-- Description list for name-value pairs -->
<dl>
    <dt>HTML</dt>
    <dd>HyperText Markup Language, the standard markup language for web pages</dd>
    
    <dt>CSS</dt>
    <dd>Cascading Style Sheets, used for describing the presentation of a document</dd>
</dl>

Screen readers announce the list type and the number of items, providing valuable context to users.

Accessible Images and Media

Images, videos, and other media need special consideration to ensure they're accessible to all users.

Alternative Text for Images

The alt attribute provides a text alternative for images, which is essential for screen reader users:

<!-- Informative image with descriptive alt text -->
<img src="chart-quarterly-sales.png" alt="Bar chart showing quarterly sales growth of 15% in Q1, 22% in Q2, 30% in Q3, and 25% in Q4">

<!-- Decorative image with empty alt text -->
<img src="decorative-divider.png" alt="">

<!-- Image that is a link needs alt text describing the destination -->
<a href="products.html">
    <img src="product-catalog.jpg" alt="View our product catalog">
</a>

Guidelines for writing effective alt text:

  • Be specific and concise (typically 125 characters or less)
  • Describe the purpose or content of the image, not just what it looks like
  • Include any text that appears in the image
  • For decorative images that don't add content, use empty alt text (alt="")
  • Don't start with "Image of" or "Picture of" (screen readers already announce it's an image)
  • For complex images like charts or graphs, provide a more detailed description nearby

Complex Images

For complex images like charts, diagrams, or infographics, a short alt text might not be sufficient. In these cases, provide a more detailed description:

<figure>
    <img src="complex-chart.png" 
         alt="Chart showing relationship between temperature and productivity" 
         aria-describedby="chart-description">
    <figcaption id="chart-description">
        This chart illustrates how workplace productivity decreases as temperature increases 
        beyond 72°F (22°C). Productivity peaks at 70-72°F and drops by approximately 
        2% for every degree above 77°F, with a sharp decline above 85°F.
    </figcaption>
</figure>

Accessible Video and Audio

Make multimedia content accessible with captions, transcripts, and audio descriptions:

<!-- Video with captions and audio description -->
<figure>
    <video controls>
        <source src="video.mp4" type="video/mp4">
        <source src="video.webm" type="video/webm">
        <track kind="captions" src="captions.vtt" srclang="en" label="English" default>
        <track kind="descriptions" src="descriptions.vtt" srclang="en" label="Audio Descriptions">
        Your browser does not support the video tag.
    </video>
    <figcaption>
        <a href="transcript.html">View transcript</a>
    </figcaption>
</figure>

<!-- Audio with transcript -->
<figure>
    <audio controls>
        <source src="podcast.mp3" type="audio/mpeg">
        <source src="podcast.ogg" type="audio/ogg">
        Your browser does not support the audio tag.
    </audio>
    <figcaption>
        <details>
            <summary>Transcript</summary>
            <p>Host: Welcome to our podcast about web accessibility...</p>
            <p>Guest: Thank you for having me...</p>
            <!-- Full transcript content -->
        </details>
    </figcaption>
</figure>

Key accessibility features for multimedia:

  • Captions: Text versions of spoken content and important sounds, synchronized with the media
  • Transcripts: Text versions of all audio content, available separately from the media
  • Audio Descriptions: Narration describing important visual details that aren't conveyed through dialogue
  • Controls: Ensure media players have accessible controls for play, pause, volume, etc.
  • No Auto-Play: Avoid auto-playing media, which can interfere with screen readers

Accessible Forms

Forms are often critical interaction points on websites, and they need special attention to ensure accessibility.

Labeling Form Controls

Every form control should have a properly associated label:

<!-- Explicit label with for attribute (preferred method) -->
<div class="form-group">
    <label for="name">Full Name</label>
    <input type="text" id="name" name="name" required>
</div>

<!-- Implicit label by wrapping the input -->
<div class="form-group">
    <label>
        Email Address
        <input type="email" name="email" required>
    </label>
</div>

<!-- Visually hidden label (still accessible to screen readers) -->
<div class="form-group">
    <label for="search" class="visually-hidden">Search the website</label>
    <input type="search" id="search" name="search" placeholder="Search...">
    <button type="submit">Search</button>
</div>

<!-- CSS for visually hidden elements -->
<style>
    .visually-hidden {
        position: absolute;
        width: 1px;
        height: 1px;
        margin: -1px;
        padding: 0;
        overflow: hidden;
        clip: rect(0, 0, 0, 0);
        border: 0;
    }
</style>

Never rely solely on placeholder text as a label, as it disappears when the user starts typing and may not be announced by all screen readers.

Grouping Related Controls

Use <fieldset> and <legend> to group related form controls:

<fieldset>
    <legend>Shipping Address</legend>
    
    <div class="form-group">
        <label for="street">Street Address</label>
        <input type="text" id="street" name="shipping_street" required>
    </div>
    
    <div class="form-group">
        <label for="city">City</label>
        <input type="text" id="city" name="shipping_city" required>
    </div>
    
    <div class="form-group">
        <label for="zip">Postal Code</label>
        <input type="text" id="zip" name="shipping_zip" required>
    </div>
</fieldset>

<fieldset>
    <legend>Preferred Contact Method</legend>
    
    <div class="form-group">
        <input type="radio" id="contact-email" name="contact_method" value="email">
        <label for="contact-email">Email</label>
    </div>
    
    <div class="form-group">
        <input type="radio" id="contact-phone" name="contact_method" value="phone">
        <label for="contact-phone">Phone</label>
    </div>
    
    <div class="form-group">
        <input type="radio" id="contact-text" name="contact_method" value="text">
        <label for="contact-text">Text Message</label>
    </div>
</fieldset>

This approach is particularly important for radio buttons and checkboxes, as it provides context about the relationship between the options.

Form Instructions and Error Messages

Provide clear instructions and accessible error messages:

<!-- Instructions before the form -->
<div class="form-instructions">
    <p>All fields marked with an asterisk (*) are required.</p>
</div>

<!-- Field-specific instructions -->
<div class="form-group">
    <label for="password">Password *</label>
    <input 
        type="password" 
        id="password" 
        name="password" 
        aria-describedby="password-requirements"
        required
    >
    <div id="password-requirements" class="field-hint">
        Password must be at least 8 characters and include a number and a special character.
    </div>
</div>

<!-- Inline error message -->
<div class="form-group has-error">
    <label for="email">Email Address *</label>
    <input 
        type="email" 
        id="email" 
        name="email" 
        aria-describedby="email-error"
        aria-invalid="true"
        required
    >
    <div id="email-error" class="error-message" role="alert">
        Please enter a valid email address.
    </div>
</div>

Key attributes for accessible form validation:

  • aria-describedby: Associates instructions or error messages with form controls
  • aria-invalid="true": Indicates that a field has an invalid value
  • role="alert": Makes error messages announced by screen readers when they appear
  • required: Indicates that a field is required

Accessible Form Controls

Use the most appropriate input types and attributes:

<!-- Use appropriate input types -->
<input type="email" id="email" name="email"> <!-- Triggers email keyboard on mobile -->
<input type="tel" id="phone" name="phone"> <!-- Triggers phone keyboard on mobile -->
<input type="number" id="quantity" name="quantity" min="1" max="10"> <!-- Numeric input with constraints -->
<input type="date" id="birthdate" name="birthdate"> <!-- Date picker -->

<!-- Provide autocomplete hints -->
<input type="text" id="name" name="name" autocomplete="name">
<input type="email" id="email" name="email" autocomplete="email">
<input type="tel" id="phone" name="phone" autocomplete="tel">

<!-- Use datalist for suggestions -->
<label for="country">Country</label>
<input type="text" id="country" name="country" list="country-list">
<datalist id="country-list">
    <option value="Australia">
    <option value="Canada">
    <option value="France">
    <option value="Germany">
    <option value="Japan">
    <option value="United Kingdom">
    <option value="United States">
</datalist>

Best Practices for Accessible Forms

  • Logical Tab Order: Ensure form controls can be navigated in a logical order using the Tab key
  • Keyboard Accessibility: All form interactions should be possible using only a keyboard
  • Error Prevention: Use client-side validation to catch errors before submission
  • Error Recovery: Provide clear error messages and focus on the first error when form submission fails
  • Sufficient Time: Avoid session timeouts or provide warnings and options to extend
  • No Reliance on Color: Don't use color alone to indicate required fields or errors
  • Large Touch Targets: Make form controls large enough to be easily tapped on touch screens

Keyboard Accessibility

Many users navigate websites using only a keyboard, including people with motor disabilities, visual impairments, or those who simply prefer keyboard navigation.

Focus Management

Ensure all interactive elements can receive keyboard focus and that focus is visible:

<!-- Ensure all interactive elements are focusable -->
<a href="page.html">Standard link (naturally focusable)</a>

<button type="button">Standard button (naturally focusable)</button>

<!-- For custom interactive elements, add tabindex="0" -->
<div tabindex="0" role="button" aria-pressed="false" onclick="togglePanel()" onkeydown="handleKeydown(event)">
    Custom button
</div>

<!-- CSS for visible focus styles -->
<style>
    /* Enhance default focus styles */
    :focus {
        outline: 3px solid #4d90fe;
        outline-offset: 2px;
    }
    
    /* Only hide outline for mouse users, keep it for keyboard users */
    :focus:not(:focus-visible) {
        outline: none;
    }
    
    :focus-visible {
        outline: 3px solid #4d90fe;
        outline-offset: 2px;
    }
</style>

<!-- JavaScript for custom keyboard interaction -->
<script>
    function handleKeydown(event) {
        // Handle Enter or Space key to activate the element
        if (event.key === 'Enter' || event.key === ' ') {
            event.preventDefault();
            togglePanel();
        }
    }
    
    function togglePanel() {
        // Toggle panel logic here
        const button = event.currentTarget;
        const isPressed = button.getAttribute('aria-pressed') === 'true';
        button.setAttribute('aria-pressed', !isPressed);
    }
</script>

Key principles for keyboard accessibility:

  • All interactive elements must be focusable and operable with a keyboard
  • Focus order should follow a logical sequence, typically matching the visual order
  • Focus should be visible at all times (don't remove focus outlines without providing alternatives)
  • Custom interactive elements should mimic native behavior (e.g., respond to Enter and Space keys)

Skip Links

Provide skip links to help keyboard users bypass repetitive navigation:

<!-- Skip links at the beginning of the page -->
<div class="skip-links">
    <a href="#main-content" class="skip-link">Skip to main content</a>
    <a href="#footer" class="skip-link">Skip to footer</a>
</div>

<header>
    <!-- Header content and navigation -->
</header>

<main id="main-content" tabindex="-1">
    <!-- Main content -->
</main>

<footer id="footer" tabindex="-1">
    <!-- Footer content -->
</footer>

<!-- CSS for skip links -->
<style>
    .skip-link {
        position: absolute;
        top: -40px;
        left: 0;
        padding: 8px;
        background-color: #fff;
        color: #000;
        z-index: 100;
        transition: top 0.3s;
    }
    
    .skip-link:focus {
        top: 0;
    }
</style>

Skip links are typically visually hidden until they receive focus, allowing keyboard users to bypass repetitive content while remaining invisible to mouse users.

Keyboard Traps

Avoid keyboard traps, where users can't navigate away from an element using the keyboard:

<!-- Bad example: Keyboard trap in a modal -->
<div class="modal" role="dialog" aria-labelledby="modal-title">
    <h2 id="modal-title">Modal Title</h2>
    <p>Modal content...</p>
    <button onclick="closeModal()">Close</button>
    <!-- No way to exit with keyboard if closeModal() doesn't work -->
</div>

<!-- Good example: Modal with keyboard accessibility -->
<div class="modal" role="dialog" aria-labelledby="modal-title" aria-modal="true">
    <h2 id="modal-title">Modal Title</h2>
    <p>Modal content...</p>
    <button onclick="closeModal()" aria-label="Close modal">Close</button>
</div>

<script>
    // When modal opens, store the element that had focus
    let previousFocus;
    
    function openModal() {
        const modal = document.querySelector('.modal');
        previousFocus = document.activeElement;
        
        // Show modal
        modal.style.display = 'block';
        
        // Focus the first focusable element in the modal
        const firstFocusable = modal.querySelector('button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])');
        firstFocusable.focus();
        
        // Add event listener for Escape key
        document.addEventListener('keydown', handleEscapeKey);
    }
    
    function closeModal() {
        const modal = document.querySelector('.modal');
        
        // Hide modal
        modal.style.display = 'none';
        
        // Restore focus to the element that had focus before the modal opened
        if (previousFocus) {
            previousFocus.focus();
        }
        
        // Remove event listener for Escape key
        document.removeEventListener('keydown', handleEscapeKey);
    }
    
    function handleEscapeKey(event) {
        if (event.key === 'Escape') {
            closeModal();
        }
    }
    
    // Trap focus within the modal
    function trapFocus(event) {
        const modal = document.querySelector('.modal');
        if (!modal.contains(event.target)) {
            event.stopPropagation();
            
            const focusableElements = modal.querySelectorAll('button, [href], input, select, textarea, [tabindex]:not([tabindex="-1"])');
            const firstElement = focusableElements[0];
            const lastElement = focusableElements[focusableElements.length - 1];
            
            if (event.shiftKey && event.target === firstElement) {
                lastElement.focus();
                event.preventDefault();
            } else if (!event.shiftKey && event.target === lastElement) {
                firstElement.focus();
                event.preventDefault();
            }
        }
    }
</script>

The only acceptable keyboard traps are those that are temporary and necessary for functionality, such as a modal dialog where focus should be trapped until the dialog is closed.

ARIA (Accessible Rich Internet Applications)

ARIA is a set of attributes that define ways to make web content and applications more accessible to people with disabilities.

ARIA Roles

ARIA roles define what an element is or does:

<!-- Landmark roles (most should use semantic HTML instead) -->
<div role="navigation">...</div> <!-- Better: <nav>...</nav> -->
<div role="main">...</div> <!-- Better: <main>...</main> -->

<!-- Widget roles -->
<div role="button" tabindex="0">Click me</div>
<div role="tab" tabindex="0" aria-selected="true">Tab 1</div>
<div role="tabpanel">Tab 1 content</div>

<!-- Document structure roles -->
<div role="heading" aria-level="2">Heading</div> <!-- Better: <h2>Heading</h2> -->
<div role="list">
    <div role="listitem">Item 1</div>
    <div role="listitem">Item 2</div>
</div> <!-- Better: <ul><li>Item 1</li><li>Item 2</li></ul> -->

Important: Always prefer native HTML elements with built-in semantics over ARIA roles when possible. ARIA should be used to enhance accessibility, not replace proper HTML.

ARIA States and Properties

ARIA states and properties provide additional information about elements:

<!-- Labels and descriptions -->
<button aria-label="Close">×</button>
<div aria-labelledby="title">...</div>
<h2 id="title">Section Title</h2>
<button aria-describedby="tooltip">Help</button>
<div id="tooltip" hidden>Click for assistance</div>

<!-- States -->
<button aria-pressed="true">Toggle</button>
<div role="checkbox" tabindex="0" aria-checked="false">Accept terms</div>
<div aria-expanded="false">Show more</div>
<input aria-invalid="true">

<!-- Relationships -->
<div role="combobox" aria-controls="dropdown" aria-expanded="false">Select</div>
<div id="dropdown" hidden>...</div>
<div aria-owns="child">Parent</div>
<div id="child">Child</div>

Common ARIA attributes and their purposes:

  • aria-label: Provides a text label for an element
  • aria-labelledby: References another element that serves as the label
  • aria-describedby: References another element that provides additional description
  • aria-expanded: Indicates if a collapsible element is expanded or collapsed
  • aria-hidden: Hides content from assistive technologies
  • aria-live: Defines an area that will be updated dynamically
  • aria-required: Indicates that user input is required
  • aria-selected: Indicates the current selection state of elements

Live Regions

ARIA live regions announce dynamic content changes to screen reader users:

<!-- Polite live region (waits for user idle) -->
<div aria-live="polite" aria-atomic="true">
    <p id="status-message">Your changes have been saved.</p>
</div>

<!-- Assertive live region (interrupts immediately) -->
<div aria-live="assertive" aria-atomic="true">
    <p id="error-message">An error occurred. Please try again.</p>
</div>

<!-- Live region with additional attributes -->
<div 
    aria-live="polite" 
    aria-atomic="true" 
    aria-relevant="additions text"
    class="notification-area">
    <!-- Dynamic content will be inserted here -->
</div>

<script>
    // Update a live region
    function showNotification(message) {
        const notificationArea = document.querySelector('.notification-area');
        
        // Clear existing content
        notificationArea.innerHTML = '';
        
        // Add new message
        const notification = document.createElement('div');
        notification.textContent = message;
        notificationArea.appendChild(notification);
        
        // Remove after 5 seconds
        setTimeout(() => {
            notification.remove();
        }, 5000);
    }
</script>

Live region attributes:

  • aria-live="polite": Announces changes when the user is idle (use for most updates)
  • aria-live="assertive": Announces changes immediately (use only for critical information)
  • aria-atomic="true": Announces the entire region when any part changes
  • aria-relevant="additions text": Specifies what types of changes should be announced

The First Rule of ARIA

The first rule of ARIA is: Don't use ARIA if you don't need to.

Native HTML elements have built-in semantics and behavior that are well-supported by browsers and assistive technologies. Only use ARIA when:

  • You're creating a custom widget that doesn't have a native HTML equivalent
  • You need to fix accessibility issues in existing code that can't be rewritten
  • You need to provide additional information that can't be conveyed through native HTML

Using ARIA incorrectly can actually make accessibility worse, not better.

Accessible Navigation and Links

Navigation is a critical part of any website, and it needs to be accessible to all users.

Accessible Navigation Menus

Create navigation menus that work for all users:

<!-- Basic accessible navigation -->
<nav aria-label="Main Navigation">
    <ul>
        <li><a href="/" aria-current="page">Home</a></li>
        <li><a href="/about">About</a></li>
        <li><a href="/services">Services</a></li>
        <li><a href="/contact">Contact</a></li>
    </ul>
</nav>

<!-- Dropdown navigation -->
<nav aria-label="Main Navigation">
    <ul>
        <li><a href="/">Home</a></li>
        <li>
            <button 
                aria-expanded="false" 
                aria-controls="services-dropdown">
                Services
                <span aria-hidden="true">▼</span>
            </button>
            <ul id="services-dropdown" hidden>
                <li><a href="/services/web-design">Web Design</a></li>
                <li><a href="/services/development">Development</a></li>
                <li><a href="/services/consulting">Consulting</a></li>
            </ul>
        </li>
        <li><a href="/about">About</a></li>
        <li><a href="/contact">Contact</a></li>
    </ul>
</nav>

<script>
    // Toggle dropdown menu
    document.querySelectorAll('nav button').forEach(button => {
        button.addEventListener('click', () => {
            const expanded = button.getAttribute('aria-expanded') === 'true';
            button.setAttribute('aria-expanded', !expanded);
            
            const dropdown = document.getElementById(button.getAttribute('aria-controls'));
            dropdown.hidden = expanded;
            
            // If opening the dropdown, set focus to the first link
            if (!expanded) {
                setTimeout(() => {
                    dropdown.querySelector('a').focus();
                }, 10);
            }
        });
    });
    
    // Close dropdown when Escape key is pressed
    document.addEventListener('keydown', event => {
        if (event.key === 'Escape') {
            document.querySelectorAll('nav button[aria-expanded="true"]').forEach(button => {
                button.click();
                button.focus();
            });
        }
    });
    
    // Close dropdown when clicking outside
    document.addEventListener('click', event => {
        if (!event.target.closest('nav li')) {
            document.querySelectorAll('nav button[aria-expanded="true"]').forEach(button => {
                button.setAttribute('aria-expanded', 'false');
                document.getElementById(button.getAttribute('aria-controls')).hidden = true;
            });
        }
    });
</script>

Key features of accessible navigation:

  • Use proper semantic structure (<nav>, <ul>, <li>)
  • Provide an aria-label for each navigation section
  • Use aria-current="page" to indicate the current page
  • For dropdowns, use aria-expanded and aria-controls to manage state
  • Ensure keyboard accessibility for all navigation interactions
  • Provide a way to close dropdowns using the Escape key

Accessible Links

Create links that provide clear information about their destination:

<!-- Good link text -->
<a href="accessibility-guidelines.html">Web Content Accessibility Guidelines</a>

<!-- Bad link text (avoid) -->
<a href="accessibility-guidelines.html">Click here</a> to read the Web Content Accessibility Guidelines.

<!-- Link with additional context for screen readers -->
<a href="report.pdf">
    Annual Report 
    <span class="file-info">(PDF, 2.5MB)</span>
</a>

<!-- Link that opens in a new window -->
<a href="https://example.com" target="_blank" rel="noopener">
    Example Website
    <span class="visually-hidden">(opens in a new window)</span>
</a>

<!-- Skip link for keyboard users -->
<a href="#main-content" class="skip-link">Skip to main content</a>

Best practices for accessible links:

  • Use descriptive link text that makes sense out of context
  • Avoid generic phrases like "click here" or "read more"
  • Indicate when links open in a new window or download files
  • Ensure sufficient color contrast between link text and background
  • Make sure links are visually distinguishable from surrounding text (not by color alone)
  • Provide skip links to help keyboard users bypass repetitive content

Testing for Accessibility

Testing is a crucial part of ensuring your website is accessible to all users.

Automated Testing Tools

Automated tools can help identify many accessibility issues:

  • WAVE: Web Accessibility Evaluation Tool by WebAIM
  • axe: Accessibility testing engine by Deque Systems
  • Lighthouse: Built into Chrome DevTools
  • NVDA: Free screen reader for Windows
  • VoiceOver: Built-in screen reader for macOS and iOS
  • Color Contrast Analyzers: Tools to check color contrast ratios

Manual Testing Checklist

Automated tools can't catch everything. Use this checklist for manual testing:

Category Test How to Test
Keyboard Accessibility Tab Navigation Navigate the entire page using only the Tab key. Ensure all interactive elements are focusable and in a logical order.
Focus Visibility Ensure focus is clearly visible on all interactive elements.
Keyboard Traps Verify that no element traps keyboard focus (except for modal dialogs).
Screen Reader Testing Content Reading Use a screen reader to navigate the page. Ensure all content is announced correctly.
Image Descriptions Verify that images have appropriate alt text that conveys their purpose.
Form Labels Ensure all form controls have properly associated labels that are announced by screen readers.
Visual Testing Color Contrast Check that text has sufficient contrast against its background (4.5:1 for normal text, 3:1 for large text).
Text Zoom Zoom text to 200% and ensure the page remains usable without horizontal scrolling.
Content Testing Heading Structure Verify that headings create a logical outline of the page content.
Link Purpose Ensure all links have descriptive text that makes sense out of context.

User Testing

The most valuable accessibility testing involves real users with disabilities:

  • Recruit users with different disabilities (visual, motor, cognitive, etc.)
  • Provide specific tasks for users to complete
  • Observe their interactions and note any difficulties
  • Ask for feedback on their experience
  • Address issues identified during testing

Accessibility Statement

Consider including an accessibility statement on your website that:

  • States your commitment to accessibility
  • Specifies which guidelines you follow (e.g., WCAG 2.1 Level AA)
  • Lists any known accessibility issues and workarounds
  • Provides contact information for users to report accessibility problems
  • Outlines your plan for ongoing accessibility improvements

An accessibility statement demonstrates your commitment to inclusivity and provides valuable information for users with disabilities.

Summary

In this lesson, we've explored HTML accessibility, which is essential for creating websites that can be used by everyone, including people with disabilities:

  • Introduction to Web Accessibility: What accessibility is, its benefits, and the different types of disabilities that affect web usage
  • Semantic HTML for Accessibility: Using proper document structure, heading hierarchy, and text-level semantics
  • Accessible Images and Media: Providing alternative text, captions, transcripts, and audio descriptions
  • Accessible Forms: Labeling form controls, grouping related fields, and providing clear instructions and error messages
  • Keyboard Accessibility: Ensuring all interactive elements can be used with a keyboard, managing focus, and avoiding keyboard traps
  • ARIA (Accessible Rich Internet Applications): Using ARIA roles, states, properties, and live regions to enhance accessibility
  • Accessible Navigation and Links: Creating navigation menus and links that work for all users
  • Testing for Accessibility: Using automated tools, manual testing, and user testing to identify and fix accessibility issues

Accessibility is not just a technical requirement but a fundamental aspect of creating an inclusive web that works for everyone. By implementing the techniques covered in this lesson, you can ensure that your websites are accessible to all users, regardless of their abilities or how they access the web.

Remember these key points:

  • Use semantic HTML as the foundation of accessibility
  • Provide text alternatives for non-text content
  • Ensure keyboard accessibility for all interactive elements
  • Create a logical structure with proper heading hierarchy
  • Label all form controls and provide clear instructions
  • Use ARIA attributes to enhance accessibility when needed
  • Test with assistive technologies to ensure a good user experience
  • Consider accessibility from the beginning of the design and development process
